# 全量问题修复与抖音云最小闭环计划

## Summary
- 目标是基于现有审计结果，制定“全量问题清单”修复路线，并将抖音云接入推进到“最小可用闭环”。
- 实施顺序采用“先阻塞后优化”：先恢复关键流程可测可跑，再推进中低优先级问题与结构优化，最后完成云端读写闭环验收。

## Current State Analysis
- 双关主线（1-10 → 1-100）整体方向存在，但在重开参数保持、二关恢复门禁、状态语义一致性上有断点。
- 集成测试存在明显阻塞：GameManager 契约漂移、AnalyticsManager 导出/调用不一致、云存储集测超时与返回契约不一致。
- 项目存在双形态维护（`game.js` 与 `src/core/*`），并伴随 CJS/ESM 混用，导致实现与测试对齐成本高。
- 云能力已具备基础：`get_open_id`、`save_progress`、`load_progress` 已存在，但客户端读取仍有直连 DB 路径，尚未收敛为统一云函数链路。

## Assumptions & Decisions
- 决策：修复范围采用“全量问题清单”，但分阶段交付，先消除阻塞再优化。
- 决策：云接入目标采用“最小闭环”，优先打通登录+读写+降级可回退，不在本轮扩展到完整上线运营能力。
- 假设：以现有 `__tests__/integration/*` 作为主要回归门槛，同时保留单测作为局部校验。
- 假设：不做破坏式重构，不引入新基础设施，仅在现有结构上做兼容修复与收敛。
