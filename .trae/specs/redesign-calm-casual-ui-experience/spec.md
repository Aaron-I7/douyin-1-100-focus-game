# 游戏界面与交互美化 Spec

## Why
当前游戏界面风格单调、信息层级与排版一致性不足，且用户反馈存在遮挡和过场感知不明显的问题。需要参考市场热门轻休闲产品（如“抓大鹅”）的视觉节奏与交互表达，在不增加眼压的前提下提升品质感与可玩性。

## What Changes
- 增加“市场风格调研”产出，基于“抓大鹅”等轻休闲头部产品沉淀可执行的视觉与交互设计准则。
- 重构游戏内视觉系统：背景层次、色板、组件样式、HUD 排版、按钮与卡片规范。
- 优化关卡过场表达：第一关完成后过场动画必须清晰可见并自然进入第二关。
- 修复并约束“遮挡”问题：HUD、安全区、可玩区布局必须无冲突。
- 统一交互节奏：保持护眼、低刺激、无震动反馈，并提供可配置的动效强度策略。

## Impact
- Affected specs: 游戏视觉设计、关卡过场、布局与安全区、交互反馈策略
- Affected code: `src/core/ui/ui-theme-system-legacy.js`、`src/core/ui/ui-render-engine-legacy.js`、`src/core/services/services-game-manager-runtime-legacy.js`、`src/core/state/state-transition-manager-legacy.js`、`src/core/ui/ui-ui-manager-legacy.js`

## ADDED Requirements
### Requirement: 市场参考驱动的视觉规范
系统 SHALL 基于市场热门轻休闲游戏的共性体验，形成可落地的“视觉与交互规范清单”并用于实现。

#### Scenario: 形成规范清单
- **WHEN** 开始界面美化实施前
- **THEN** 必须先产出并确认包含色彩、排版、层级、动效节奏、反馈策略的规范清单

### Requirement: 轻休闲风格可执行视觉交互规范清单
系统 SHALL 将“抓大鹅、羊了个羊、跳一跳”等样本中的轻休闲共性转化为可直接实施与验收的清单。

#### Scenario: 执行视觉交互清单
- **WHEN** 进入视觉重构与交互实现阶段
- **THEN** 所有界面与交互实现必须满足以下清单项，未达标项不得进入验收

#### Checklist: 市场样本共性拆解（抓大鹅等）
| 维度 | 可执行规范 | 验收标准 |
|---|---|---|
| 风格基调 | 采用“低饱和暖色 + 高可读深色文字”主配色，禁止荧光高饱和大面积铺底 | 主界面主色饱和度中低，标题与正文文本对比清晰可读 |
| 信息层级 | 界面固定三层：背景层、功能层、反馈层；同屏层级不超过 3 个视觉焦点 | 任一主要页面可明确识别 1 个主焦点与 2 个次焦点 |
| 卡片与按钮 | 组件统一圆角（8/12/16 分档）与轻阴影，不使用硬边高对比描边 | 主要按钮、弹窗、HUD 卡片视觉风格一致且状态可辨 |
| 文案密度 | 单屏高频信息不超过 5 组，优先短词与数字化反馈 | 对局页无大段说明文；关键状态一眼可读 |
| HUD 布局 | 顶部展示目标与进度，底部展示操作入口，中央保留完整可玩区 | HUD 与可点击格子无重叠，横竖比变化时仍可点击 |
| 色彩反馈 | 正确/错误/警示仅使用固定语义色并限制动画时长，避免频闪 | 连续点击场景无强闪烁，状态变化在 300ms 内可感知 |
| 动效节奏 | 采用“短反馈、弱位移、低振幅”策略，单次反馈建议 120~260ms | 点击反馈立即可见且不拖沓，无连续眩晕或跳闪 |
| 过场表现 | 关卡切换采用遮罩淡入淡出 + 文案提示，避免复杂镜头运动 | 第一关结束后过场完整可见且自动衔接到第二关 |
| 触控体验 | 点击热区最小 48x48，误触区域与高频点击区域分离 | 高密度点击场景误触率可控，无“点不到/点偏”问题 |
| 听觉与震动 | 默认关闭震动与强刺激音效，可保留轻提示音且可关闭 | 默认配置下无震动触发，反馈仍可被视觉识别 |

#### Checklist: 交付与验收产物
| 产物 | 最低要求 | 验收方式 |
|---|---|---|
| 样本拆解表 | 至少覆盖抓大鹅与 2 个同类轻休闲样本，按页面/组件/反馈拆解 | 文档走查确认字段完整 |
| 视觉 Token 表 | 包含颜色、圆角、间距、层级、动效时长五类 Token | 设计稿与实现参数逐项对照 |
| 页面对照稿 | 提供改造前后对照图（首页、对局页、结算页） | 评审会逐页核对差异与一致性 |
| 交互验收单 | 包含点击反馈、过场动画、安全区、误触保护检查项 | QA 按清单逐条勾检并留档 |

### Requirement: 无遮挡布局与安全区适配
系统 SHALL 保证 HUD、可玩区、底部进度区域在不同屏幕下不互相遮挡。

#### Scenario: 竖屏常见机型
- **WHEN** 进入对局并渲染 HUD 与格子区
- **THEN** 顶部信息区与格子区域必须分离，且底部进度条不压住可点击格子

### Requirement: 第一关完成过场可感知
系统 SHALL 在第一关完成后展示明确可感知的过场动画，并自动切换到第二关。

#### Scenario: 第一关通关
- **WHEN** 玩家完成第一关最后一个数字
- **THEN** 显示过场动画（含文案与视觉过渡）并在动画结束后自动开始第二关

## MODIFIED Requirements
### Requirement: 护眼交互反馈
系统 SHALL 默认关闭震动与高刺激反馈，同时保留低刺激视觉层次，避免界面“纯白无层次”。

#### Scenario: 对局点击反馈
- **WHEN** 玩家点击正确或错误数字
- **THEN** 不触发震动与强刺激特效，界面依然通过低对比层次保持状态可辨识

## REMOVED Requirements
### Requirement: 强刺激赛博装饰反馈
**Reason**: 与“护眼低刺激”目标冲突，且降低可读性与长时体验。
**Migration**: 使用低对比背景层次、轻量过场与信息层级替代高亮闪烁/强发光/震动反馈。
